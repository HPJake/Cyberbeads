# 2026-06-24 缩放下限 & 大网格性能优化

## 缩放下限 0.5 → 0.2

### 改动
`js/app.js` 两处 `clamp` 下限从 `0.5` 改为 `0.2`：
- 滚轮缩放 (line ~439)
- 双指捏合缩放 (line ~464)

### 理由
大网格 (50×50+) 在默认视口下需要缩小更多才能看到全景。20% 缩放让用户有更大调整空间。

## 大网格流畅度优化：占用格集合 + 绘制帧率节流

之前的"逐格增量重绘"方案因为 `clearRect` 擦到相邻豆子边缘（半径 19.5px ≈ 半格 20px）导致渲染残缺，已被回退。本次采用两个安全方案，完全不碰 `_drawBeadAt`、`drawIronedFlat` 等绘制方法。

### 方案 A：占用格集合（Occupancy Set）

**原理**：维护 `GridModel.occupiedCells` (`Set<"r,c">`)，`drawAllBeads()` 遍历这个 Set 而非全部 `rows × cols`。

**改动：**

- `js/config.js`：无新增常量
- `js/grid-model.js`：
  - 构造函数新增 `this.occupiedCells = new Set()`
  - `placeBead()` → `this.occupiedCells.add(key)`
  - `removeBead()` → `this.occupiedCells.delete(key)`
  - `clear()` → `this.occupiedCells.clear()`
  - `resize()` / `load()` → 调用 `_rebuildOccupiedSet()`
  - `undo()` / `redo()` → 成功后调用 `_rebuildOccupiedSet()`
  - 新增 `_rebuildOccupiedSet()` 全量扫描重建
  - `hasAnyBead()` 改为 `return this.occupiedCells.size > 0`（O(1) 替代 O(n²)）
- `js/renderer.js`：
  - `drawAllBeads()` 改为 `for (const key of this.grid.occupiedCells)` 迭代，手动 `indexOf(',')` 解析 r,c（避免 `split` 分配临时数组）

**效果**：100×100 网格 50% 填充 → 10,000 次遍历 → 5,000 次，越稀疏效果越好。

### 方案 B：绘制帧率节流

**原理**：非熨烫模式下，`drawAllBeads()` 最低间隔 30ms（~33fps），减少多余的重绘。

**改动：**

- `js/config.js`：新增 `BEAD_DRAW_INTERVAL: 30`
- `js/app.js`：
  - 构造函数新增 `this.lastBeadDrawTime = 0`
  - 游戏循环 bead 绘制段：熨烫模式间隔 0ms（全帧率），否则用 `BEAD_DRAW_INTERVAL` 节流
  - 未到时间的帧跳过绘制但保持 `beadDirty = true`，下帧重试，确保最后一帧不丢失

**效果**：画笔拖动画豆子时，重绘频率从 ~83Hz 降到 ~33Hz。人眼看不出差异。

### 总效果

| 场景 | 原遍历次数 | 优化后 |
| --- | --- | --- |
| 100×100 50% 填充 + 拖动绘画 | 10,000 × ~83fps | 5,000 × ~33fps |
| 100×100 10% 填充 + 拖动绘画 | 10,000 × ~83fps | 1,000 × ~33fps |

两个方案都不修改豆子绘制逻辑，零渲染品质风险。
